# Monitoring

## Что мониторим в API

- `bank_reco_request_total`: количество запросов по endpoint'ам.
- `bank_reco_error_total`: число ошибок API. Рост метрики сигнализирует о проблемах в сервисе или схеме входных данных.
- `bank_reco_latency_seconds`: latency по endpoint'ам. Для `POST /predict` это основной технический SLA.
- `bank_reco_score_distribution`: распределение score рекомендаций. Просадка в низкие значения часто означает ухудшение качества или дрейф входных данных.
- `bank_reco_empty_total`: доля запросов без выданных рекомендаций. Нужна, чтобы заметить случаи, когда клиенту нечего предложить или модель деградирует.
- `bank_reco_suspicious_total`: запросы с признаками data drift или редких категорий относительно обучающей выборки.

## Базовые ML-метрики для офлайн-контроля

- `Precision@3`: насколько часто рекомендации в top-3 действительно становятся новыми продуктами.
- `Recall@3`: сколько реальных новых продуктов удаётся покрыть в top-3.
- `MAP@3`: основная ranking-метрика проекта, потому что учитывает и попадание, и порядок рекомендаций.
- `NDCG@3`: полезна как дополнительная ranking-метрика, потому что сильнее поощряет верный порядок в начале списка.

## Что считать drift-сигналом

- Числовые признаки выходят за тренировочный диапазон `p01-p99` слишком часто.
- Категориальные признаки приходят с новыми или очень редкими значениями.
- Растёт доля пустых рекомендаций.
- Score распределение смещается вниз.

В коде API уже есть базовая проверка числовых и категориальных признаков через `reference_stats.json`, который сохраняется после обучения.

## Когда запускать дообучение

- По расписанию раз в месяц после появления нового среза клиентов.
- Внепланово, если несколько дней подряд:
  - растёт `bank_reco_suspicious_total`;
  - ухудшается онлайн-конверсия рекомендаций;
  - заметно падают офлайн ranking-метрики на свежем бэктесте;
  - меняется продуктовая линейка или появляются новые каналы/сегменты.

## Практический минимум сопровождения

- Еженедельно проверять latency и error rate.
- После каждого нового месячного среза пересчитывать офлайн-метрики на последнем holdout-периоде.
- Обновлять `reference_stats.json` вместе с новой моделью.
